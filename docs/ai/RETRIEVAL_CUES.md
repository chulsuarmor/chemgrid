# 인출 단서 (Retrieval Cues)
> 사용자가 특정 업무 구조/방법론에 고유 이름을 부여한 목록입니다.
> 해당 단어가 사용자에게서 나오면 이 파일의 정의를 즉시 참조하십시오.
> 이 파일은 모든 하네스 세션에서 공유됩니다.
>
> **원리:** 사용자가 임의의 단어(예: "울가락")를 명명하면, 해당 단어는 맥락적으로 자연 발생할 수 없으므로 AI가 할루시네이션으로 지어낼 수 없습니다. 반드시 이 파일을 참조해야만 정확한 의미를 파악할 수 있습니다.

## 1. 하네스 오버 하네스 (Harness-over-Harness)

**명명일:** 2026-03 초
**정의:** 에이전트 오케스트레이션을 기반으로 각기 전문화된 에이전트에게 업무 직렬과 부서를 부여하고, 업무 종류별로 부서를 개설하여 최종 관리자 에이전트(CT)로부터 하달받은 명령을 하위 에이전트들부터 차례로 작업→검수→감사 등의 과정을 거쳐 단계적으로 상신하는 업무 구조.

**세부 원칙:**
1. CT(최종관리자)는 직접 구현 절대 금지 — 방향성 하달과 감사 확인만 수행
2. 3계층 위임: CT → MM(중간관리자) → Worker(실무자)
3. 각 에이전트는 skills/mistakes MD를 축적하며 세션마다 진화
4. 감사팀은 CT 직속 상위부서로서 산출물 품질 + CT 월권까지 감사
5. PLC(전문적학습공동체)로 직렬 간 교훈 공유
6. Ralph Loop로 사용자 부재 시에도 자동 피드백 루프 가동

**상세 참조:** `docs/ai/HARNESS_OVER_HARNESS_REFERENCE.md`

## 2. 공회전 (하네스 서큘레이션 테스트)

**명명일:** 2026-03-19
**정의:** 하네스 오버 하네스 구조가 실제로 작동하는지 검증하는 사전 테스트. 실제 업무를 수행하기 전에 각 계층(CT/MM/Worker/감사팀)이 skills/mistakes 기반으로 전문 업무를 수행하고, 부적합한 것은 반려하며, 완벽할 때까지 반복하는 검증 프로세스.

**핵심 검증 항목:**
1. Worker가 skills를 읽고 이해한 흔적이 있는가
2. MM이 Worker의 skills 이해도를 실제로 검수하는가
3. 감사팀이 부적합 산출물을 실제로 반려하는가
4. CT가 직접 구현하지 않고 위임만 하는가
5. skills/mistakes가 실제로 갱신되는가

**공회전 통과 전 실제 업무 수행 금지.**

**최초 실행:** 2026-03-19, AI Office. 2개 도메인(기숙사+화학) 교차 검증 → PASS 확정.

**상세 참조:** `docs/ai/HARNESS_OVER_HARNESS_REFERENCE.md` §9

## 3. Ralph Loop (랄프 루프)

**명명일:** 2026-03 초
**정의:** 사용자가 자는 동안에도 CT가 master_plan.md를 읽고 → 각 도메인 PENDING 확인 → MM spawn → 감사 확인 → 문서 업데이트 → 세션 종료(컨텍스트 클리어) → 외부 bash 루프가 재실행하는 자동 피드백 루프.

**핵심:** 한 사이클이 끝나면 반드시 세션을 종료하여 컨텍스트를 클리어. 기억이 가득 차면 환각 발생.

**상세 참조:** `CLAUDE.md` Rule 10 루프 1~4단계

---

## 4. 하이브리드 위임 (Hybrid Delegation)

**명명일:** 2026-03-19
**정의:** CT가 Agent spawn(MM/감사팀)과 Bash+claude CLI(Worker)를 병행하는 위임 구조. Worker를 `claude --print --model claude-opus-4-6`로 완전 독립 프로세스로 실행하여 fresh context + 전체 도구 + Agent 중첩까지 가능하게 함.

**검증:** 2026-03-19 테스트 — CLI Worker가 skills 읽기, Agent 도구 사용 가능, Chrome MCP 접근 가능 확인.

**상세 참조:** `docs/ai/HARNESS_OVER_HARNESS_REFERENCE.md` §9.7
