[1] 프로젝트 개요
================================================================================

ChemGrid는 PyQt6 기반 화학 구조 작성 + 분석 통합 플랫폼이다.
사용자가 분자를 그리면 Lewis 구조, 이론적 구조, ESP 전자구름, 스펙트럼 예측,
3D 시각화, 반응 메커니즘, 역합성, 도킹까지 하나의 앱에서 수행한다.
최종 목표는 AlphaFold 단백질 예측과 결합하여 신약 개발 보조 시스템을 구축하는 것.

기술 스택: Python 3.12 / PyQt6 / RDKit / ORCA 6.1.1 / AutoDock Vina
개발 환경: Conda env 'chemgrid' on Windows 11

[2] 전체 구조도: Harness-over-Harness v2 (5-Layer Multi-Agent System)
================================================================================

이 프로젝트의 핵심 혁신은 "AI가 AI를 관리하는" 다층 하네스 구조이다.

    ┌─────────────────────────────────────────────────┐
    │           사용자 (Human Supervisor)               │
    │     최종 의사결정, 방향성 하달, 절차 감독          │
    └────────────────────┬────────────────────────────┘
                         │ 지시/피드백
    ┌────────────────────▼────────────────────────────┐
    │         CT (Control Tower / 컨트롤 타워)          │
    │   - master_plan.md로 전체 프로젝트 통제           │
    │   - 직접 코딩 절대 금지                          │
    │   - 하위 에이전트 spawn + 결과 검토              │
    │   - Cascade 단위로 업무 일괄 디스패치             │
    └────┬──────────┬──────────┬──────────┬───────────┘
         │          │          │          │
    ┌────▼───┐ ┌───▼────┐ ┌──▼───┐ ┌───▼──────┐
    │최종감사 │ │전문감사 │ │QA감사│ │시각피드백 │  ← Layer 2: 감사/검수 계층
    │(Final) │ │(×3팀)  │ │(×3명)│ │감사관     │
    └────────┘ └────────┘ └──────┘ └──────────┘
         │          │          │          │
    ┌────▼──────────▼──────────▼──────────▼───────────┐
    │              12개 실무 부서 (Department)           │  ← Layer 3: 부서 계층
    │  ui_canvas / chem_engine / rendering / 3d_viewer  │
    │  spectroscopy / reaction_synthesis / dft_orca     │
    │  docking / export_integration / visual_feedback   │
    │  testing_build / alphafold_drug                   │
    └────┬──────────┬──────────┬──────────────────────┘
         │          │          │
    ┌────▼───┐ ┌───▼────┐ ┌──▼─────┐
    │  MM    │ │   P    │ │   R    │  ← Layer 4: 부서 내 3-에이전트
    │중간관리│ │기획자  │ │검수자  │
    │(관리)  │ │(구현)  │ │(검증)  │
    └────────┘ └────────┘ └────────┘

각 부서는 3명의 에이전트로 구성된다:
- MM (Middle Manager): 태스크 분해 + P/R spawn + 결과 검토. 직접 코딩 금지.
- P (Planner/Implementer): 실제 코드 작성. 할당된 OWNED_FILES만 수정.
- R (Reviewer): 코드 검증(py_compile, ast.parse, headless test). 코드 수정 금지.

[3] 하네스(Harness) 지침 요청 방식
================================================================================

"하네스"란 AI 에이전트의 행동 범위를 제한하는 틀을 의미한다.
이 프로젝트에서 하네스는 다음 문서들로 구현된다:

[3-1] 정적 하네스 (파일 기반)
- CLAUDE.md (루트): 전체 AI 행동 규칙 10조
- departments/ARCHITECTURE.md: 5-Layer 구조, 통신 프로토콜, 트리거 규칙
- departments/dept_XXX/CLAUDE.md: 부서별 역할, OWNED_FILES, 의존성, 특별 주의사항
- docs/ai/mistakes.md: 과거 실수 DB → "같은 실수를 반복하지 않도록"
- docs/ai/skills/: 도메인별 작업 매뉴얼 (스킬 파일)

[3-2] 동적 하네스 (세션 간 기억)
- master_plan.md: CT가 관리하는 전체 프로젝트 상태 + 피드백 하달용
- context_list.md: 부서별 태스크 체크리스트 (PENDING/COMPLETED)
- context_note.md: 부서별 기술적 판단/에러 해결 기록 (시방서)
- MEMORY.md + memory/: Claude Code 자동 기억 시스템

[3-3] 지침 요청 흐름 (Cascade 방식)
1. 사용자 → CT: "이 기능 구현해"
2. CT → master_plan.md: 태스크 기록 + 부서별 할당
3. CT → Agent spawn: 해당 부서 MM을 깨움 (부서 CLAUDE.md 경로 + 태스크 전달)
4. MM → P spawn: 기획자에게 구현 위임
5. P → 코드 작성 + _source/ 동기화
6. MM → R spawn: 검수자에게 검증 위임
7. R → PASS/FAIL 판정
8. PASS → MM → SUBMIT 보고서 작성 → 감사팀으로 상신
9. 감사팀 → 전문 검증 (화학 데이터 교차 참조) → PASS/WARN/FAIL
10. PASS → 최종감사 → 실제 GUI 실행 + 스크린샷 검증
11. 전체 PASS → CT 보고 → 전학공 → 다음 Cascade
